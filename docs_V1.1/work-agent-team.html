核心亮点Key takeaways

  • 契约驱动并行开发：架构 Agent 产出 4 份契约，后端 / 前端 / 数据 Agent 凭契约独立并行推进，互不读对方代码。Contract-driven parallel development: architect produces 4 contracts; backend, frontend, and data agents work in parallel without reading each other's code.
  • 8 层质量门禁：单元测试、集成测试、类型检查、代码风格、安全扫描、性能回归、契约校验、视觉一致性——全自动。8-layer quality gates: unit tests, integration tests, type checking, linting, security scan, performance regression, contract validation, visual consistency — fully automated.
  • 测试 Agent 自修复内循环：门禁不过先自己修，最多 3 次整体重试。功能缺陷才升级仲裁，日常失败不打扰人类。Self-healing test loop: test agent auto-fixes before escalating. Only functional defects require human arbitration.
  • 人类只做三件事：拍板、仲裁、验收。日常开发由 Orchestrator 调度，人类仅在关键决策点介入。Humans do three things: decide, arbitrate, accept. Orchestrator handles everything else.
01

背景与理念Background & philosophy

用 AI Agent 写软件，最大的瓶颈不是模型能力，而是上下文爆炸和质量控制。单个 Agent 承担所有角色——需求、架构、前端、后端、测试——很快上下文就会爆炸，质量无法保证。这套框架把软件工程拆成 8 个专职 Sub-Agent，通过 4 份契约文件（API / 数据库 / 前端 / 设计系统）实现并行开发，通过 8 层质量门禁保障产出质量。

The bottleneck in AI software engineering isn't model capability — it's context explosion and quality control. When one agent handles requirements, architecture, frontend, backend, and testing, context inevitably overflows and quality degrades. This framework splits work across 8 specialized sub-agents, enabling parallel development via 4 contract files, and enforces quality through 8-layer quality gates.

8 层质量门禁8-layer quality gates

测试 Agent 在合并产物上跑 8 层门禁，全部通过后才交运维 Agent 部署。任何一层失败，先由测试 Agent 自修复（最多 3 次整体重试），功能缺陷才升级为仲裁。After code merge, the test agent runs all 8 layers. Any failure triggers auto-fix first (max 3 full retries); only functional defects escalate to arbitration.

开发工作流Development workflow

一套完整的开发流程——从需求到交付，人类只需在关键节点做决策：A complete development pipeline — humans only decide at key checkpoints:

  1. 内容ContentPRD + 合规自检compliance
    将模糊想法转化为结构清晰的 PRD 和合规自检清单。From vague ideas to structured PRD with compliance checklist.
  2. 架构Architecture4 份契约contracts
    产出架构文档 + OpenAPI + DB Schema + Frontend Spec + Design System。Produces architecture doc + OpenAPI + DB schema + frontend spec + design system.
  3. 后端 ∥ 前端 ∥ 数据Backend ∥ Frontend ∥ Data并行Parallel
    凭契约独立并行开发，互不读对方代码。发现契约缺陷走变更通道，先做无争议部分。Parallel development on contracts. Contract issues go through change channels — non-controversial parts proceed first.
  4. 运维 · 合并Ops · MergeGit
    合并三路代码，解决冲突（业务冲突上报主人）。Merge all branches; escalate business conflicts only.
  5. 测试Test8 层门禁gates
    8 层门禁 + 内循环自修复（最多 3 次）。全绿 → 部署；3 次不过 → 仲裁。8-layer gates + self-healing (max 3 retries). All green → deploy; 3 failures → arbitration.
  6. 运维 · 部署Ops · Deploy灰度→全量Canary→full
    灰度部署 → 健康检查 → 全量 → 监控 → 产品健康简报回流给内容 Agent。Canary deploy → health check → full rollout → monitoring → health report loops back to content agent.
05

关键机制Key mechanisms

契约变更逆向通道Contract change reverse channel

开发者发现契约缺字段/漏端点时，不私改契约，而是走"变更申请 → 架构 Agent 仲裁 → 广播新版契约"通路。申请期间不阻塞——先做无争议部分，收到新版契约后再补争议部分。

When agents find contract gaps, they file change requests instead of modifying contracts unilaterally. Architecture agent arbitrates, then broadcasts new versions. Non-controversial work continues during arbitration.

渐进扩容策略Progressive scaling

不是一上来就上满 9 个 Agent。按功能复杂度触发扩容：启蒙期 5 Agent（合体实现）→ 成长期 6 Agent（拆出前端）→ 成熟期 8 Agent（拆出后端+数据）。每加一个 Agent 多一份调度成本，在"上下文爆炸"成为现实痛点之前不拆。

Don't deploy all 9 agents from day one. Scale by complexity: early phase 5 (combined implementation) → growth phase 6 (split frontend) → mature phase 8 (split backend + data). Each agent adds coordination cost — only split when context explosion becomes a real bottleneck.

熔断机制Circuit breaker

同一功能变更超过 5 次，强制升级仲裁——防止 Agent 在错误方向上反复修正消耗 token。

Same feature change >5 times triggers mandatory arbitration — preventing agents from burning tokens on cyclic corrections.
